# High-Load Analytics Ingestion Service

Высокопроизводительный сервис асинхронного сбора (ingestion) аналитических событий на Go. Спроектирован для обработки пиковых нагрузок, параллельной обработки данных без потерь (**Zero Data Loss**) и эффективной записи в базу данных с помощью интеллектуального батчинга.

---

## 🚀 Производительность

Нагрузочное тестирование проводилось с помощью `hey` при постоянной высокой нагрузке (50–100 параллельных клиентов в течение 30 секунд).

| Метрика | Значение |
|---------|----------|
| **Throughput** | **~14 000 RPS** |
| **Средняя задержка** | **3.6 ms** |
| **Потери данных** | **0%** (обработано и сохранено более **500 000+** событий) |
| **CPU Profile** | Около **40%** времени процессора занимают системные network I/O вызовы, что указывает на отсутствие узких мест в приложении и минимальный lock contention |

## 🏗️ Архитектура

Проект реализован по принципам **Clean Architecture** и оптимизирован под высокий throughput.

```text
                  Client
                     │
                     ▼
          Rate Limiter Middleware
                     │
        ┌────────────┴────────────┐
    (Запрос ок)              (Лимит превышен)
        │                         │
        ▼                         ▼
 HTTP Handler (Chi)          429 Too Many Requests
        │
        ▼
 Buffered Go Channel
        │
        ▼
 Background Workers
        │
 Batch Flush (size/time)
        │
        ▼
 PostgreSQL
```

### Асинхронная обработка

HTTP-хендлер валидирует входящий JSON, помещает событие в буферизированный Go-канал и сразу возвращает клиенту `202 Accepted`.

За счёт этого клиент не ожидает выполнения операций записи в базу данных, а сетевой поток освобождается практически мгновенно.

### Батчинг

Фоновые воркеры постоянно считывают события из канала и выполняют **bulk insert** в PostgreSQL.

Запись выполняется:

- при достижении заданного размера батча;
- либо по таймеру каждые **5 секунд**.

Такой подход значительно снижает количество SQL-запросов и уменьшает нагрузку на пул соединений PostgreSQL.

### Rate Limiting

Используется собственная middleware с поддержкой in-memory/Redis-хранилища.

Для корректной идентификации клиентов применяется `net.SplitHostPort`, что исключает ложные срабатывания из-за изменения клиентских портов под высокой параллельной нагрузкой.

Используется стратегия **Fail-Open** — если сервис лимитирования временно недоступен, запросы продолжают обслуживаться.

### Graceful Shutdown

При получении сигналов `SIGINT` или `SIGTERM` приложение:

1. прекращает принимать новые HTTP-запросы;
2. дожидается завершения работы фоновых воркеров;
3. записывает оставшиеся события из памяти в PostgreSQL;
4. корректно закрывает подключения к базе данных и Redis;
5. завершает процесс с кодом `0`.

Это гарантирует отсутствие потери данных даже при остановке контейнера.

## 🛠️ Стек технологий

| Технология | Назначение |
| :--- | :--- |
| **Go (Golang)** | Основной язык, высококонкурентная среда выполнения |
| **Chi Router** | Легковесный, идиоматичный HTTP-роутер |
| **PostgreSQL** | Реляционная СУБД для постоянного хранения аналитических данных |
| **Redis** | Высокопроизводительное In-Memory хранилище для отслеживания лимитов запросов (Rate Limiting) |
| **Log/Slog** | Структурированное, высокопроизводительное JSON-логирование |
| **Docker / Compose** | Контейнеризация приложения и оркестрация всей инфраструктуры |
| **Pprof** | Профилирование CPU и памяти в рантайме под нагрузкой |

## ⚙️ Локальный запуск

### Требования

- Docker
- Docker Compose

### Запуск

```bash
docker compose up --build -d
```

Будут автоматически запущены:

- Go API (`:8080`)
- Redis
- PostgreSQL
- SQL-миграции

---

## Проверка работы

```bash
curl -X POST http://localhost:8080/event \
-H "Content-Type: application/json" \
-d '{
  "user_id": 1,
  "event_type": "page_view",
  "time": "2026-07-02T12:00:00Z",
  "page_url": "http://localhost/home"
}'
```

Ожидаемый ответ:


```text
202 Accepted
```
💡 Запрос мгновенно возвращает ответ, после чего событие асинхронно отправляется в буфер для последующей записи в БД.

## 🧪 Проверка Graceful Shutdown

Во время выполнения нагрузочного теста остановите контейнер приложения:

```bash
docker compose stop analytics-app
```

Затем изучите логи:

```bash
docker compose logs analytics-app
```

В логах будет видно, что приложение:

- корректно обработало сигнал завершения;
- завершило приём новых запросов;
- дождалось окончания работы воркеров;
- записало оставшиеся события в PostgreSQL;
- завершилось без паники и без потери данных.
