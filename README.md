# High-Load Analytics Ingestion Service

Высокопроизводительный сервис асинхронного сбора (ingestion) аналитических событий на Go. Спроектирован для обработки пиковых нагрузок, параллельной обработки данных без потерь (**Zero Data Loss**) и эффективной записи в базу данных с помощью интеллектуального батчинга.

---

## 🚀 Производительность

Нагрузочное тестирование проводилось с помощью `hey` при постоянной высокой нагрузке (50–100 параллельных клиентов в течение 30 секунд).

| Метрика | Значение |
|---------|----------|
| **Throughput** | **16 217+ RPS** |
| **Средняя задержка (p50)** | **2.9 ms** |
| **Хвостовая задержка (p99)** | **6.9 ms** |
| **Потери данных** | **0%** (обработано и успешно сохранено **486 556** событий за 30 сек) |
| **CPU Profile (Network I/O)** | **38.25%** времени процессора занимают системные вызовы `linux.Syscall6`, что доказывает отсутствие lock contention, низкий оверхед рантайма и фокус на сетевом вводе-выводе. |

## 🏗️ Архитектура

Проект реализован по принципам **Clean Architecture** и оптимизирован под высокий throughput.

```text
                  Client
                     │
                     ▼
          Reverse Proxy (Nginx/LB) -> [Извлечение X-Forwarded-For / X-Real-IP]
                     │
                     ▼
          Rate Limiter Middleware (Redis / Lua script)
                     │
        ┌────────────┴────────────┐
    (Запрос ок)              (Лимит превышен)
        │                         │
        ▼                         ▼
 HTTP Handler (Chi)          429 Too Many Requests
        │
        ▼
 Buffered Go Channel
        │
        ▼
 Background Workers (Worker Pool)
        │
 ┌──────┴────────────────────────────────────────┐
 │   Batch Flush Trigger (Size or Interval)      │
 │                                               │
 │   [Успех] -> Конец                            │
 │   [Ошибка сети] -> Exponential Backoff Retry  │
 │   [Poison Pill] -> Fallback: Построчный Изолятор│
 └──────┬────────────────────────────────────────┘
        │
        ▼
   PostgreSQL (pgx.CopyFrom)
```

## ✅ Настоящий Zero Data Loss & Отказоустойчивость

В отличие от базовых реализаций асинхронного батчинга, данный сервис защищён от потери данных на всех этапах транзиентных сбоев:

- **Экспоненциальный Backoff Retry** — при возникновении временных проблем с подключением к СУБД (сетевые моргания, таймауты) воркер не уничтожает батч, а выполняет серию повторных попыток записи с увеличивающимся интервалом ожидания (`500ms -> 1000ms -> 2000ms`).
- **Fallback-изоляция («Ядовитые строки»)** — метод `pgx.CopyFrom` атомарен: одна невалидная строка (например, превышение длины `VARCHAR`) может забраковать весь батч из 500 элементов. В случае ошибки структуры данных сервис автоматически переходит в построчный режим вставки (fallback). Невалидное событие логируется отдельно и изолируется, а остальные 499 валидных событий успешно сохраняются в БД.
- **Безопасность горутин** — каждая операция сброса ограничена жёстким таймаутом `context.WithTimeout`, что предотвращает утечки горутин и бесконечные блокировки воркеров при дедлоках в СУБД.

## ⚡ Асинхронная обработка и батчинг

HTTP-хендлер валидирует входящий JSON, помещает событие в буферизированный Go-канал и сразу возвращает клиенту `202 Accepted`, мгновенно высвобождая сетевой поток.

Фоновые воркеры считывают события и упаковывают их в пачки. Сброс батча (Flush) в PostgreSQL через высокопроизводительный протокол `CopyFrom` триггерится по двум условиям: при достижении `BATCH_SIZE` элементов или по тикеру `FLUSH_INTERVAL`.

## 🛡️ Rate Limiting за Reverse Proxy

Middleware лимитера адаптирована для работы в реальном продакшене за балансировщиками нагрузки (Nginx, Envoy, Cloudflare). Определение IP-адреса клиента учитывает заголовки `X-Forwarded-For` и `X-Real-IP`, предотвращая ложное блокирование самого прокси-сервера. Лимитер использует стратегию **Fail-Open** — при недоступности Redis сервис продолжает бесперебойно принимать события.


### Graceful Shutdown

При получении сигналов `SIGINT` или `SIGTERM` приложение:

1. прекращает принимать новые HTTP-запросы;
2. дожидается завершения работы фоновых воркеров;
3. записывает оставшиеся события из памяти в PostgreSQL;
4. корректно закрывает подключения к базе данных и Redis;
5. завершает процесс с кодом `0`.

Это гарантирует отсутствие потери данных даже при остановке контейнера.

## 🛠️ Стек технологий

| Технология | Назначение |
| :--- | :--- |
| **Go (Golang)** | Основной язык, высококонкурентная среда выполнения |
| **Chi Router** | Легковесный, идиоматичный HTTP-роутер |
| **PostgreSQL** | Реляционная СУБД для постоянного хранения аналитических данных |

| **Log/Slog** | Структурированное, высокопроизводительное JSON-логирование |
| **Docker / Compose** | Контейнеризация приложения и оркестрация всей инфраструктуры |
| **Pprof** | Профилирование CPU и памяти в рантайме под нагрузкой |

```text

```

## 🧪 Проверка Graceful Shutdown

Во время выполнения нагрузочного теста остановите контейнер приложения:

```bash
docker compose stop analytics-app
```

Затем изучите логи:

```bash
docker compose logs analytics-app
```

В логах будет видно, что приложение:

- корректно обработало сигнал завершения;
- завершило приём новых запросов;
- дождалось окончания работы воркеров;
- записало оставшиеся события в PostgreSQL;
- завершилось без паники и без потери данных.
