# High-Load Analytics Ingestion Service

Асинхронный сервис сбора аналитических событий на Go. Спроектирован для высокого throughput на приём (HTTP → буфер → `202 Accepted` за миллисекунды) и эффективной записи в PostgreSQL через батчинг и `COPY`.

## 🏗️ Архитектура

```text
                  Client
                     │
                     ▼
          Reverse Proxy (Nginx/LB) -> [Извлечение X-Forwarded-For / X-Real-IP]
                     │
                     ▼
          Rate Limiter Middleware (Redis / Lua script)
                     │
        ┌────────────┴────────────┐
    (Запрос ок)              (Лимит превышен)
        │                         │
        ▼                         ▼
 HTTP Handler (Chi)          429 Too Many Requests
        │
        ▼
 Buffered Go Channel
        │
        ▼
 Background Workers (Worker Pool)
        │
 ┌──────┴────────────────────────────────────────┐
 │   Batch Flush Trigger (Size or Interval)      │
 │                                               │
 │   [Успех] -> Конец                            │
 │   [Ошибка сети] -> Exponential Backoff Retry  │
 │   [Poison Pill] -> Fallback: Построчный Изолятор│
 └──────┬────────────────────────────────────────┘
        │
        ▼
   PostgreSQL (pgx.CopyFrom)
```

Слои разделены по Clean-Architecture-подобной схеме: `handler` (HTTP) → `service` (бизнес-логика, батчинг) → `repository` (доступ к БД), с доменными типами и интерфейсами, объявленными на стороне потребителя.

## ✅ Надёжность доставки

Сервис устойчив к кратковременным сбоям, но **не гарантирует zero data loss** при длительной недоступности БД — это осознанный компромисс между надёжностью и простотой (полная гарантия потребовала бы WAL/спилла на диск или внешней очереди).

- **Экспоненциальный backoff retry** — при временных проблемах с подключением к СУБД (сетевые моргания, короткие таймауты) воркер не отбрасывает батч сразу, а выполняет до 3 повторных попыток записи с увеличивающимся интервалом (`500ms → 1000ms → 2000ms`).
- **Fallback-изоляция «ядовитых строк»** — `pgx.CopyFrom` атомарен: одна невалидная строка может забраковать весь батч из 500 элементов. При ошибке батча сервис переходит в построчный режим вставки; невалидное событие логируется и отбрасывается, остальные сохраняются.
- **Таймауты на каждую попытку** — каждый вызов `InsertBatch` ограничен `context.WithTimeout`, что предотвращает утечки горутин и бесконечные блокировки при зависании СУБД.
- **Явный предел потерь** — если все 3 попытки исчерпаны (например, БД недоступна дольше нескольких секунд), батч отбрасывается с `CRITICAL`-логом. Это видимая, а не тихая потеря данных — событие остаётся в логах для последующего разбора.
- **Backpressure вместо блокировки** — если буферный канал переполнен (воркеры не успевают), `Create` немедленно возвращает `503 Service Unavailable`, а не блокирует HTTP-хендлер.

## ⚡ Асинхронная обработка и батчинг

HTTP-хендлер валидирует входящий JSON, помещает событие в буферизированный Go-канал и сразу возвращает клиенту `202 Accepted`, мгновенно высвобождая сетевой поток.

Пул из `WORKER_COUNT` фоновых воркеров считывает события и упаковывает их в пачки. Сброс батча (flush) в PostgreSQL через `CopyFrom` триггерится по двум условиям: при достижении `BATCH_SIZE` элементов или по тикеру `FLUSH_INTERVAL` — в зависимости от того, что наступит раньше.

## 🛡️ Rate Limiting

Middleware использует Redis (Lua-скрипт с `INCR`/`PEXPIRE`) для скользящего окна лимита запросов на клиента. Стратегия — **fail-open**: при недоступности Redis сервис продолжает принимать события без ограничения, приоритет отдан приёму данных.

Определение клиента учитывает заголовки `X-Forwarded-For` / `X-Real-IP`.

> ⚠️ **Требование к развёртыванию:** эти заголовки читаются безусловно, без allowlist доверенных прокси. Сервис необходимо разворачивать строго за reverse proxy, который сам выставляет/перезаписывает `X-Forwarded-For` — иначе клиент может подделать заголовок и обойти лимит. Дефолтный `CLIENT_LIMIT=1000000` за `RATE_LIMIT_WINDOW=60s` рассчитан на нагрузочное тестирование; для продакшена значение нужно снизить под реальный трафик.

## 🔄 Graceful Shutdown

При получении `SIGINT` или `SIGTERM` приложение:

1. прекращает принимать новые HTTP-запросы (`http.Server.Shutdown`);
2. дожидается завершения работы фоновых воркеров;
3. записывает оставшиеся события из буфера в PostgreSQL;
4. корректно закрывает подключения к Redis и базе данных;
5. завершает процесс с кодом `0`.

## ⚙️ Конфигурация

Все параметры задаются через переменные окружения (см. `.env`):

| Переменная | По умолчанию | Назначение |
| :--- | :--- | :--- |
| `HTTP_PORT` | `8080` | Порт HTTP API |
| `DB_HOST`, `DB_PORT`, `DB_USER`, `DB_PASSWORD`, `DB_NAME` | — (обязательны) | Подключение к PostgreSQL |
| `DB_SSLMODE` | `disable` | Режим SSL для подключения к БД |
| `REDIS_ADDRESS` | — (обязателен) | Адрес Redis для rate limiter'а |
| `WORKER_COUNT` | `5` | Число фоновых воркеров, пишущих батчи в БД |
| `CHAN_SIZE` | `10000` | Размер буферного канала между HTTP-хендлером и воркерами |
| `BATCH_SIZE` | `500` | Размер батча, при котором происходит принудительный flush |
| `FLUSH_INTERVAL` | `1s` | Интервал, по истечении которого батч сбрасывается, даже если не набрал `BATCH_SIZE` |
| `CLIENT_LIMIT` | `1000000` | Лимит запросов на клиента за `RATE_LIMIT_WINDOW` |
| `RATE_LIMIT_WINDOW` | `60s` | Окно rate limiter'а |

> `.env` в этом репозитории содержит dev-значения для локального запуска. Для продакшена используйте секрет-менеджер вместо файла в git.

## 🧪 Тестирование и диагностика

### Запуск юнит-тестов

```bash
go test -v -race ./...
```

Покрыто тестами:

- **`internal/service`** — жизненный цикл воркеров (flush по размеру и по тикеру), graceful drain при shutdown, backpressure (`ErrQueueFull`), retry-логика при транзиентных и постоянных ошибках, идемпотентность `Stop`, гонка `Create`/`Stop` при завершении работы;
- **`internal/domain`** — граничные случаи валидации события (`Event.Validate`);
- **`internal/handler`** — маппинг доменных ошибок в HTTP-статусы (`statusFromErr`);
- **`internal/repository/postgres`** — маппинг ошибок PostgreSQL в доменные (`mapError`).

Не покрыто юнит-тестами (требует интеграционного окружения): fallback построчной вставки в реальную БД, поведение Redis-лимитера.

### Проверка работы API

```bash
curl -X POST http://localhost:8080/event \
-H "Content-Type: application/json" \
-d '{
  "user_id": 1,
  "event_type": "page_view",
  "time": "2026-07-02T12:00:00Z",
  "page_url": "http://localhost/home"
}'
```

Ожидаемый ответ: `202 Accepted`. Запрос мгновенно возвращает ответ, после чего событие асинхронно уходит в буфер для последующей записи в БД.

#### Коды ответов

| Код | Когда |
| :--- | :--- |
| `202 Accepted` | Событие принято в буфер |
| `400 Bad Request` | Невалидный JSON, неизвестное поле, не прошла доменная валидация (`Event.Validate`) |
| `429 Too Many Requests` | Превышен лимит запросов для клиента |
| `503 Service Unavailable` | Буферный канал переполнен — воркеры не успевают вычитывать; клиенту следует повторить запрос позже |

## 🧪 Проверка Graceful Shutdown

Во время выполнения нагрузочного теста остановите контейнер приложения:

```bash
docker compose stop analytics-app
```

Затем изучите логи:

```bash
docker compose logs analytics-app
```

В логах будет видно, что приложение:

- корректно обработало сигнал завершения;
- завершило приём новых запросов;
- дождалось окончания работы воркеров;
- записало оставшиеся события в PostgreSQL;
- завершилось без паники и без потери данных из буфера, накопленного на момент сигнала.
